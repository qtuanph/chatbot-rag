# 2.2 — Chat Workflow

Luồng chat hiện tại là **stateless**.

## Tóm tắt

```text
Client gửi messages gần nhất
-> backend resolve tenant
-> retrieve context theo tenant
-> gọi LLM
-> stream/trả response
```

## Hai mặt chat

### Chat nội bộ webapp

- route: `POST /api/v1/chat/stream`
- auth bằng JWT nội bộ
- `platform_admin` có thể chọn tenant để test
- `tenant_admin` chỉ chat trong tenant của mình

### Public API

- prefix: `/api/v1/public/v1`
- route chính:
  - `GET /public/v1/health`
  - `GET /public/v1/models`
  - `POST /public/v1/chat/completions`
- auth bằng tenant API key

## Stateless rule

Backend không giữ transcript bền:

- không restore history
- không lưu session chat như core product flow
- frontend chỉ giữ một cửa sổ messages ngắn trong memory

## Retrieval rule

Retrieval phải lọc theo `tenant_id`:

1. embed query
2. hybrid search trong Qdrant
3. rerank
4. assemble context
5. gọi LLM

## Provider rule

| Loại | Mặc định hiện tại |
|---|---|
| Embedding | Docker Model Runner |
| Reranker | NVIDIA NIM |
| LLM | 9Router |

## Response rule

### Internal chat

- dùng SSE
- frontend nhận chunk
- chunk cuối mang citations + usage

### Public chat

- hỗ trợ non-stream và stream
- shape gần OpenAI-compatible

## Cache / tốc độ

Project có cache cho retrieval/usage, nhưng không có persisted history.

Điểm cần nhớ:

- chậm ở chat không phải vì load session cũ
- chậm thường nằm ở retrieval, rerank hoặc model provider

## Invariants

- mọi chat phải có tenant context
- `platform_admin` thiếu `tenant_id` khi test thì backend phải chặn
- browser business request không gọi backend trực tiếp, phải qua `/api/bep/*`
- không lưu transcript vào `localStorage`
