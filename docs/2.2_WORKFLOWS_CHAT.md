# 2.2 — Chat Workflow

Luồng chat hiện tại là **stateless chat** + **structure-aware retrieval**.

## Tóm tắt

```text
Client gửi recent messages
-> backend resolve tenant
-> normalize query (loại bỏ stopword, xử lý dấu câu, giữ nguyên ERP phrase)
-> check Semantic Cache (tenant-isolated)
-> retrieve context theo tenant
-> gọi LLM
-> stream hoặc trả response hoàn chỉnh
```

## Hai mặt chat

### Chat nội bộ webapp

- route: `POST /api/v1/chat/stream`
- auth bằng JWT nội bộ
- `platform_admin` phải chọn tenant khi test
- `tenant_admin` chỉ chat trong tenant của mình
- khi mở chat mới, frontend có thể seed một lời chào UI-only từ `tenant_setting.welcome_message`; lời chào này không được gửi vào retrieval query history

### Public API

- prefix: `/api/v1/public/v1`
- route chính:
  - `GET /public/v1/health`
  - `GET /public/v1/models`
  - `POST /public/v1/chat/completions`
- auth bằng tenant API key
- có thể bật `thinking_mode`

## Stateless rule

Backend không lưu transcript bền:

- không restore history
- không có `chat_sessions` / `chat_messages`
- frontend chỉ giữ một cửa sổ message ngắn trong memory

## Retrieval rule

Retrieval phải lọc nghiêm ngặt theo `tenant_id`.

### Query rule

- retrieval mặc định chỉ dùng **latest user query**
- history vẫn được giữ cho LLM context
- history không còn là nguồn RAG expansion mặc định

### Retrieval path

1. normalize query hiện tại (giữ nguyên thuật ngữ ERP, xoá stopword, xử lý dấu câu)
2. kiểm tra Semantic Cache (được chia theo `tenant_id` qua TagField), nếu hit cache -> trả kết quả ngay
3. nếu query mang dáng heading / numbered section -> ưu tiên section route
4. semantic route luôn truy vấn chunk collection
5. `RecursiveRetriever` mở rộng section -> chunk
5. `AutoMergingRetriever` gộp chunk về parent section khi phù hợp
6. thay sentence hit bằng local window context
7. rerank sau candidate cleanup
8. hydrate full section từ PostgreSQL cho top node nếu cần
9. gọi LLM

### Safety rule

- nếu ngữ cảnh truy xuất không xác nhận đủ menu / nút / bước -> chatbot phải trả lời theo hướng “chưa đủ căn cứ”
- không được bịa quy trình
- faithfulness verification vẫn chạy như guardrail cuối

## Provider rule

| Loại | Mặc định hiện tại |
|---|---|
| Embedding | Docker Model Runner |
| Reranker | NVIDIA NIM |
| LLM | 9Router |

## Response rule

### Internal chat

- dùng SSE
- stream trả token/delta theo thời gian thực, không buffer toàn bộ câu trả lời rồi mới đẩy ra
- chunk cuối mang citations và usage

### Public chat

- hỗ trợ non-stream và stream
- shape bám OpenAI-compatible contract hiện tại
- stream path ưu tiên độ trễ thấp; nếu hậu kiểm grounding phát hiện lệch so với ngữ cảnh thì backend ghi log debug để tiếp tục siết retrieval/prompt, không rewrite giữa chừng luồng stream

## Feedback rule

- route nội bộ: `POST /api/v1/chat/feedback`
- frontend gửi:
  - `query_text`
  - `assistant_answer`
  - `citations`
  - `feedback_type`
- backend tự gắn:
  - `tenant_id`
  - `user_id`
  - `llm_model`
  - `embedding_model`
  - `reranker_model`

## Invariants

- mọi chat phải có tenant context
- `platform_admin` thiếu `tenant_id` khi test thì backend phải chặn
- browser business request không gọi backend trực tiếp; phải qua `/api/bep/*`
- không lưu transcript vào `localStorage`
