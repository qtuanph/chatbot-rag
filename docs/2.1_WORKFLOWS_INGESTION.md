# 2.1 — Ingestion Workflow

Luồng biến file upload thành dữ liệu có thể retrieval được.

## Tóm tắt

```text
Upload -> enqueue task -> parse -> store sections -> embed -> index Qdrant -> finalize
```

## Luồng chính

1. `POST /api/v1/upload`
2. backend lưu file vào RustFS
3. tạo row trong `documents`
4. enqueue `parse_document_task`
5. worker đọc file
6. parse nội dung
7. lưu `document_sections`
8. embed + ghi vào Qdrant
9. update document sang `ready`

## Thành phần chính

| Thành phần | File |
|---|---|
| Route upload | `app/modules/documents/router.py` |
| Celery task | `app/modules/documents/tasks.py` |
| Orchestration service | `app/modules/documents/ingestion/ingestion_service.py` |
| Embed/index pipeline | `app/modules/documents/ingestion/pipeline.py` |

## Parse strategy

### File text / markdown

- parse local
- không cần gọi OCR cloud

### File phức tạp

- đi qua parser tương ứng
- rồi chuẩn hóa thành node/section

## Store strategy

### PostgreSQL

Lưu:

- `documents`
- `document_sections`

### Qdrant

Lưu:

- dense vector
- sparse/hybrid metadata
- payload có `tenant_id`, `document_id`, `section_id`

## Tenant rule

Mọi bước ingest phải giữ `tenant_id` xuyên suốt:

- upload document
- store sections
- vector payload
- duplicate detection
- retrieval về sau

## Vì sao ingest đang chậm

Hiện trạng code:

- worker ingest chạy `node-ingestion` với `solo`
- trong `run_ingestion_pipeline()` đang dùng `batch_size = 1`
- tức là ingest ưu tiên an toàn/stable hơn throughput

### Kết luận

Nếu thấy upload xong embed lâu thì đó là hành vi đúng với cấu hình hiện tại, không phải cảm giác sai.

## Realtime hiện tại

Phần tiến độ document **chưa phải realtime push**.

Hiện tại:

- backend có lưu `status`, `stage`, `progress_percent`
- frontend trang tài liệu poll lại danh sách khoảng mỗi `4` giây

Nghĩa là:

- không cần F5 tay nữa
- nhưng vẫn là polling, chưa phải SSE/WebSocket progress

## Rechunk

Có hỗ trợ rechunk:

- đọc lại markdown/artifact đã có
- xóa sections/vector cũ
- parse lại
- embed/index lại

Luồng này không nhất thiết phải đi lại toàn bộ quy trình upload ban đầu.

## Invariants

- upload không block user chờ parse xong
- `tenant_id` phải đi xuyên suốt
- không double-splitting lung tung ngoài pipeline chính
- xóa dữ liệu cũ trước khi ghi lại trong rechunk
- document status phải phản ánh đúng tiến độ thực tế
